## 0. 한 줄 요약
매일 인스타 발행을 **창작이 아니라 로테이션**으로 굴려, 유틸리티 훅으로 스크롤을 멈추고
CTA로 **집토리에 착지**시키는 반자동 파이프라인을 만든다. (완전 자동/예약 발행은 비목표 — §6.)

## 1. 배경 / 현재 구현 제약 (rev.2 정정)

### 자산
- **카드 생성기 5종**: `trade_top`·`value`·`compare`·`budget_choice`·`lifestyle`
  (`scripts/insta_cards/series/`, 지역·넛지 파라미터 구동).
- **게시 경로(현행)**: `python -m scripts.insta_cards.instagram {slug}` **한 명령이 원격 자산 검증 후 즉시
  캐러셀을 게시**한다(`instagram/api.py:publish_carousel`). 자산이 200/jpeg/8MB 이내가 아니면 게시 전에
  실패시키는 가드(`verify_assets`)가 이미 있다.
- **랜딩**: `/content/[slug]` + `GET /api/content`.
- **계측(현행)**: `content_view`는 **`utm_source`·`utm_campaign`만** 수집(`ContentViewLogger.tsx` 허용키).

### 반드시 인지할 현재 제약 (초안 PRD가 틀렸던 부분)
- **예약/초안 발행·GitHub Actions 자동 업로드는 기존 설계의 비목표**(graph-api-design §비목표).
  → "컨테이너 생성 후 나중에 사람이 승인해 발행"하는 흐름은 **현재 없다.** 게시는 즉시 실행된다.
- **`--publish`는 배포가 아니다.** 저장소 파일(posts.json·커버·IG 자산·content_index)을 갱신하고
  "커밋 필요" 상태로 끝난다(`cli.py`). 인스타 게시는 **원격(배포된) 자산 URL**을 읽으므로,
  게시 전에 **커밋 → 머지 → Cloudflare 배포 완료**가 선행돼야 한다.
- **거래 데이터에 "신고일" 컬럼이 없다.** 조회는 **계약일**(`make_date(deal_year,month,day)`) 기준이고,
  신선도 판단은 **적재일**(`created_at`) 하나뿐이다(`datasources.py`). → "어제 신고된 거래"는 현 스키마로
  정확히 표현 불가(§7-2).

## 2. 목표 / KPI
- **1차 KPI(트래픽)**: 인스타 → 집토리 **유입 세션 수** + **유입 세션의 2차 행동률**
  (착지 후 `detail_view`·`dashboard_arrival`·넛지 재실행 중 1개 이상 — 2026-07-23 PRD와 동일 지표).
- **2차 KPI(운영 지속성)**: **계획 대비 무결 발행률**, 발행당 **사람 개입 시간 중앙값 ≤ 5분**.
- KPI 계산식·귀속은 §9-8.

## 3. 비목표
- 팔로워 수 자체 극대화(감성·밈 바이럴).
- **완전 무인 발행·예약 발행**(기존 Graph API 비목표 유지; 사람 검수 게이트 필수).
- 릴스·영상, 스토리 인사이트 API.
- 새 디자인 시스템/리브랜딩.

## 5. 로테이션 설계

### 5-1. 두 축
요일 = 고정 포맷, 사이클 = 회전 지역. 발행일 → `rotation.yaml` → `(series, region/segment, nudge)` 결정.

### 5-2. 재등장 간격 — 정의 분리 (rev.2 정정)
초안의 "4~6주 재등장"은 두 지표를 혼동한 값이라 아래로 분리·재정의한다.
- **전체 피드 기준, 특정 지역이 (포맷 무관) 다시 등장하는 간격**: 하루 1개 다른 지역이면
  ≈ (전체 지역 풀 크기)일.
- **동일 `(포맷, 지역)` 조합 재등장 간격**: 포맷이 주 1회면 = (해당 포맷의 지역 풀 크기)주.
  → 반복감의 실질 기준은 이 값. **목표: 동일 (포맷,지역) 재등장 ≥ 8주** ⇒ **포맷별 지역 풀 ≥ 8개**로 설정.
  (30개면 30주로 과하게 성김. 초기엔 8~12개로 시작, 피로 시 확장.)
- **확정: 초기 지역 풀 = 수도권 우선**(서울+경기 핵심). 비수도권은 Phase 3 튜닝 때 확장.

### 5-3. 요일 배치 — Phase별 (rev.2: 주말 포맷 갭 해소)
Phase 2 신규 포맷 개발 전에도 주 7회를 채우도록 **주말 대체 포맷을 명시**한다.

| 요일 | Phase 0·1 (기존 5종만) | Phase 2+ |
|------|------------------------|----------|
| 월 | `trade_top` (지역 사이클) | 동일 |
| 화 | `value` (넛지×지역) | 동일 |
| 수 | `compare` (동네쌍) | 동일 |
| 목 | `lifestyle` (넛지 회전) | 동일 |
| 금 | `budget_choice` (예산대×지역) | 동일 |
| 토 | `trade_top` **최근 7일 변형**(다른 지역) | `daily_trade` |
| 일 | **주간 베스트 재발행** 또는 `value` 2회차 | `ratio_top` |

## 6. 게시 파이프라인 (rev.2 — 현행 구현에 맞춤)

예약/드래프트가 없으므로 "승인"은 **사람이 배포된 결과를 검수한 뒤 즉시-게시 명령을 실행**하는 것으로 정의한다.

```
1) 생성   : (로컬 or 수동 트리거) rotation → series 러너로 캐러셀·랜딩 파일 생성
2) 반영   : 생성 파일을 브랜치로 커밋 → PR → 사람 검수 → 머지        ← "--publish"는 여기까지(배포 아님)
3) 배포   : 머지로 Cloudflare(웹)/Railway(백엔드) 배포 → 원격 자산 200 도달
4) 검수   : 배포된 랜딩·커버·캡션 확인 (사람, 5분)                    ← 승인 게이트
5) 게시   : `python -m scripts.insta_cards.instagram {slug}` 실행 → verify_assets(200 강제) → 즉시 게시
6) 리마인드: (선택) 프로필 링크 갱신 / 스토리 링크 스티커
```

### 6-1. 확정 필요 운영 규칙
- **생성 반영 브랜치/머지**: 초기엔 `content/ig-YYYYMMDD` 브랜치 + PR + **사람 머지**(자동 머지 금지).
- **배포 완료 확인**: 5단계 `verify_assets`가 원격 200/jpeg/8MB를 강제 → 이것이 배포-완료 gate.
  (별도 헬스체크 불요 — 200 아니면 게시 자체가 실패.)
- **승인 만료 / 미승인**: 콘텐츠는 발행일 고정. 당일 미승인 시 **그날 slot skip** + 폴백(§9-3).
  다음 날 밀어내지 않음(피드 날짜 왜곡 방지).
- **반려·수정**: 캡션 문제 → 플레인 텍스트 오버라이드로 재게시. 데이터/선정 문제 → rotation 재생성(같은
  slug 재사용, §9-4 멱등).
- **GitHub Actions 범위(Phase 1)**: **생성·PR 초안까지만** 자동. 게시(5단계)는 사람 실행 유지
  (자동 업로드는 기존 비목표 — 파일럿 후 재검토).

## 7. 데이터 계약

### 7-1. 결정론 범위 (rev.2 정정)
"같은 날짜+config면 같은 결과"는 **데이터가 늦게 적재되면 깨진다.** 계약을 아래로 좁힌다.
- **멱등 manifest 고정**: 특정 slug의 **최초 성공 생성 결과를 manifest로 확정**하고, 재실행 시
  데이터를 다시 조회하지 않고 **manifest를 재사용**한다. → 재실행/재배포가 결과를 바꾸지 않음.
- 즉 "결정론"은 "config→결과"가 아니라 **"확정된 manifest→결과"**로 정의한다.

### 7-2. daily_trade 선행 확인 (rev.2 — 착수 차단 조건)
- 현 스키마엔 **신고일 컬럼이 없다.** "어제 신고된 거래"는 표현 불가.
- **Phase 2 착수 전 필수 확인**: (a) 원천 API/적재 파이프라인에 신고일이 있는지, 없으면
  (b) `created_at`(적재일)을 "어제 신규 노출된 거래" 프록시로 쓸지 — 이 경우 캡션·훅을 **"어제 새로
  포착된 거래(적재 기준)"**로 정직하게 표기. 확인 전에는 `daily_trade`를 캘린더에 넣지 않는다(§5-3 주말 대체 유지).

## 8. CTA / 측정 설계 (rev.2 — 계측 계약 정합)

### 8-1. UTM 계약
- **현행 로거는 `utm_source`·`utm_campaign`만 수집.** 초안의 `?from=ig&fmt=` 는 폐기.
- **목표 계약**: `utm_source=instagram&utm_medium=social&utm_campaign={series}&utm_content={slug|placement}`.
- **구현 필요 항목(명시)**: `utm_medium`·`utm_content`를 쓰려면 `ContentViewLogger` 허용키 확장 +
  이벤트 스키마 반영이 선행돼야 함. **Phase 0·1은 `utm_source`+`utm_campaign={series}`만으로 시작**,
  medium/content는 로거 확장(별도 작업) 완료 후 도입.

### 8-2. 유입 경로 운영 정의 (rev.2 신설 · 확정)
- 피드 캡션의 URL은 **클릭 불가**(인스타 정책). 실제 클릭 경로 = **프로필 링크**.
  캡션은 이미 프로필 링크 안내 + 랜딩 URL을 함께 출력(`caption.py`).
- **확정: 프로필 링크 = 고정 `/content` 인덱스.** 매일 갱신하지 않는다.
  - 그날 콘텐츠는 인덱스 최상단(최신순)에 노출되므로 별도 slug 링크 갱신 불필요
    → **5분 개입 예산에서 링크 작업 제외**, 갱신 실패 리스크·폴백 문제 없음.
  - 채널 구분 계측(프로필/스토리/DM)은 `utm_content` 도입(로거 확장) 이후로 보류 —
    그 전에는 프로필 링크 단일 채널만 신뢰.
  - (후속 옵션) 직행이 필요하면 `/content/today → 최신 slug` 리다이렉트를 추가 검토(소규모 구현).

## 9. 수용 기준 / 운영 정책 (rev.2 신설)

1. **"무결 발행" 정의**: (a) 랜딩 배포 200, (b) IG 자산 200/jpeg/8MB, (c) 포맷별 CTA/프로필 링크 정상,
   (d) 인스타 게시 성공 — **네 가지 모두 충족**해야 1건 무결로 집계.
2. **`rotation.yaml` 스키마**: `timezone`(기본 `Asia/Seoul`), `weekday_format`(월~일→series),
   `region_cycles`(series별 지역/세그 리스트), `nudge_cycles`. **기준일**: 발행일(KST 00:00).
   **날짜→인덱스**: `idx = (발행일 - 앵커일).days`; 지역 = `cycle[idx % len(cycle)]`(결정론, 랜덤 금지).
3. **데이터 부족 정책**: 대상 지역 후보<필요 개수면 → (1) 같은 사이클의 **다음 후보 지역**으로 대체,
   (2) 그래도 부족하면 **그날 skip + 주간 베스트 재발행**, (3) 알림. 조용한 축소 금지.
4. **중복 방지(멱등)**: 같은 날짜 재실행 시 **동일 slug** 재사용 + 게시 로그(`append_log` 선례)로
   **이미 게시된 slug 재게시 차단**. 랜딩도 upsert(중복 생성 금지).
5. **알림/담당/SLA (확정)**: 파일럿 규모 — 담당=본인, 알림=`daily` 러너/CLI 실패 출력·로그 수동 확인,
   SLA=당일 best-effort. (규모 확대 시 Slack 웹훅 등으로 승격.)
6. **`daily_trade` 선정식**(Phase 2, §7-2 확인 후): "눈에 띄는" 정의를 명시적 식으로
   (예: 지역 중위 대비 편차 상위 N, 최소 표본, 이상치 컷). 확정 전 착수 금지.
7. **`ratio_top` 계산식**(Phase 2): 전세가율 = 전세보증금/매매가(동일 단지·면적 매칭 규칙 명시),
   갭 = 매매가−전세가. **최소 표본 수**, 이상치(초저가·특수거래) 제외 규칙 포함.
8. **KPI 계산식**: 유입 세션 = `utm_source=instagram` content_view 세션. 2차 행동률 = 그 세션 중
   후속 이벤트 1+ 발생 비율. **세션 귀속 기간**·**중복 제거(동일 anon-key/일자)** 규칙 명시.
9. **Phase 진입/중단 기준**(수치): 아래 §10.

## 10. 단계별 실행 + 게이트 (rev.2 — 실험 설계 정정)

- **Phase 0 — 수동 파일럿(2주, 14회)**
  - **목적을 "운영 가능성·계측 정상성 검증"으로 한정**(콘텐츠 성과 판정 아님).
  - 성과 관찰은 **포맷 단위만**. 요일 효과는 표본 부족으로 **판단하지 않음**(요일 분석은 4~6주+ 또는
    포맷 교차 배치 후).
  - **Phase 1 진입 기준(수치)**: 계획 14회 중 **무결 발행 14/14**, 개입 시간 **중앙값 ≤ 5분**,
    **UTM 누락 0건**, 게시 실패 0건.
- **Phase 1 — 반자동(생성·PR 자동, 게시 사람)**: `daily` 러너 + `rotation.yaml` + GH Actions(생성·PR까지).
  진입은 Phase 0 게이트 충족 시.
- **Phase 2 — 신규 포맷**: `daily_trade`(§7-2 데이터 확인 통과 시에만)·`ratio_top`(§9-7 식 확정 시).
- **Phase 3 — 튜닝**: 4~6주+ 데이터로 요일 배치·지역 풀·CTA 규칙 최적화(요일 효과는 이 단계에서 판정).
- **중단 기준**: 2주 연속 무결 발행률 <80% 또는 개입 중앙값 >10분 → 자동화/포맷 재설계 후 재개.

## 11. 리스크 / 완화
- **배포-게시 순서 오류**: 자산 배포 전 게시 시도 → `verify_assets`가 200 미달 시 실패시킴(안전망 존재).
  운영 규칙(§6)으로 순서 보장.
- **데이터 신선도/신고일 부재**: §7-2 확인 전 `daily_trade` 금지, 캡션 정직 표기.
- **반복감**: §5-2 동일(포맷,지역) ≥8주, 풀 확장.
- **잘못된 데이터 자동 발행**: 사람 검수 게이트(§6 4단계) 유지.
- **개인정보/저작권**: 실거래·단지명은 공공데이터. 외부 저작 이미지만 주의.

## 12. 우선순위 요약
1. **Phase 0**(수동 파일럿 + 계측 정상성) — 가장 싸고, 운영 가능성부터 확인.
2. **로거 UTM 확장 여부 결정**(utm_medium/content) — 채널 구분 계측의 선행 조건.
3. **Phase 1**(daily 러너 + 생성·PR 자동) — 지속성 확보.
4. **Phase 2 신규 포맷** — `daily_trade`는 §7-2 데이터 확인이 착수 차단 조건.
