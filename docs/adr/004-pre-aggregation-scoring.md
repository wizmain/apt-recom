# ADR-004: 시설 스코어링 Pre-aggregation 전략

- **상태**: Accepted
- **날짜**: 2026-03-20

## 맥락

넛지 스코어링 API는 아파트별 주변 시설 거리를 기반으로 점수를 계산한다. 원본 `apt_facility_mapping` 테이블은 4,900만 행으로, 매 요청마다 집계하면 응답 시간이 수십 초에 달한다.

## 결정

`apt_facility_summary` 요약 테이블을 사전 생성하여, 스코어링 API는 이 테이블만 쿼리한다.

## 요약 테이블 구조

```sql
apt_facility_summary (pnu, facility_subtype)
  ├── nearest_distance_m   -- 최근접 거리
  ├── count_1km            -- 1km 이내 시설 수
  ├── count_3km            -- 3km 이내 시설 수
  └── count_5km            -- 5km 이내 시설 수
```

- 약 131,139행 (16,755 아파트 x ~8 시설유형)
- 스코어링 쿼리 대상이 4,900만 → 13만으로 99.7% 감소.

## 근거

- **성능**: 4,900만 행 실시간 집계는 비현실적. 요약 테이블로 밀리초 단위 응답 가능.
- **정확성**: 빌드 타임에 정확한 거리 기반 집계를 수행하므로 결과가 일관적이다.
- **단순성**: 스코어링 로직(`scoring.py`)이 요약 테이블의 `nearest_distance_m`만 읽으면 된다.

## 트레이드오프

- 시설 데이터 변경 시 요약 테이블을 재빌드해야 한다.
- 개별 시설 상세 조회(상세 모달)는 여전히 원본 매핑 테이블을 PNU 기준으로 쿼리한다.

## 결과

- 넛지 스코어링 API 응답 시간: 수십 초 → 수백 밀리초.
- 점수 계산: `distance_to_score()` 함수가 `nearest_distance_m`을 0~100 점수로 변환.
- 라이프 항목별 가중합으로 최종 점수 산출.
